# ADR 0011: End-to-End Encryption Key Management

## Status

Accepted (design; implementation deferred to extension phase)

## Context

ADR 0010 establishes the E2EE threat model and the decision to use the Signal Protocol (X3DH + Double Ratchet). This ADR addresses the operational questions:

1. How are keys generated, stored, and rotated?
2. What does the server store, and what does it never see?
3. How does a new device join an existing account?
4. How are keys recovered after device loss?

## Key Types and Ownership

| Key Type | Generated By | Stored On | Server Stores | Notes |
|----------|-------------|-----------|---------------|-------|
| Identity Key Pair (IK) | Client | Device (secure enclave) | Public key only | Long-lived; never rotated (changing IK = new identity) |
| Signed Pre-Key (SPK) | Client | Device + backup | Public key + signature | Rotated every 7 days; server keeps last 3 for in-flight sessions |
| One-Time Pre-Key (OPK) | Client | Device; batch uploaded | Public key | Consumed once per session; server holds ~100 at a time |
| Ephemeral Key (EK) | Client (sender) | Not stored | Not stored | Disposable; derived per session during X3DH |
| Message Keys | Double Ratchet | Not stored | Not stored | Derived in-session; never leave the device |

## Key Distribution API

### Upload Pre-Key Bundle (device registration)

```
POST /api/keys/upload
Authorization: Bearer <token>
{
  "identity_key": "<base64 IK public>",
  "signed_pre_key": {
    "key_id": 1,
    "public_key": "<base64>",
    "signature": "<base64 IK-over-SPK>"
  },
  "one_time_pre_keys": [
    { "key_id": 101, "public_key": "<base64>" },
    ...
  ]
}
```

Server stores in `device_keys` table:
```sql
CREATE TABLE device_keys (
  device_id    UUID REFERENCES devices(id),
  ik_public    BYTEA NOT NULL,
  spk_id       INT NOT NULL,
  spk_public   BYTEA NOT NULL,
  spk_sig      BYTEA NOT NULL,
  created_at   TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY (device_id)
);

CREATE TABLE one_time_pre_keys (
  device_id    UUID REFERENCES devices(id),
  key_id       INT NOT NULL,
  public_key   BYTEA NOT NULL,
  consumed_at  TIMESTAMPTZ,
  PRIMARY KEY (device_id, key_id)
);
```

### Fetch Pre-Key Bundle (initiate session)

```
GET /api/keys/bundle/{user_id}
Authorization: Bearer <token>
```

Response includes: IK, latest SPK + signature, one unused OPK (atomically marked consumed), and the target device list. Server atomically marks the OPK as consumed.

### OPK Replenishment

When OPK count drops below 10, the server sends a `keys.replenish` WS event. The client uploads a new batch of 100 OPKs.

## Multi-Device Key Management

Each device has an independent identity key pair. When Alice has two devices (phone + laptop):

1. Both devices register their own IK, SPK, OPK batches.
2. Bob fetching Alice's pre-key bundle receives bundles for **all** of Alice's devices.
3. Bob's client initiates an X3DH session with **each device independently** and sends the message ciphertext once per device session.
4. Server stores `N_devices` ciphertext copies in `message_ciphertext_per_device` table.

This is the Signal multi-device model (also used by WhatsApp). It means message delivery cost is O(devices per user), not O(1).

## Key Backup and Recovery

**Option A: No backup (Signal default)**
- Lost device = lost message history.
- Strongest security: no server-side key material ever.
- Users must accept this risk.

**Option B: Encrypted backup to cloud (WhatsApp model)**
- Client encrypts the local key store with a backup key derived from a PIN.
- Encrypted backup stored in S3 (user-owned cloud storage or EchoLine's S3).
- Recovery: enter PIN on new device → decrypt backup → restore key state.
- Risk: PIN brute-force if backup is stolen (mitigated by HSM-backed rate limiting on PIN attempts, as WhatsApp does).

**Decision: Implement Option B for registered accounts; Option A for anonymous accounts.**

## Signed Pre-Key Rotation

1. Client generates new SPK, signs it with IK, uploads to server.
2. Server stores new SPK, keeps the previous SPK for 7 days (for in-flight sessions still using it).
3. After 7 days, server deletes old SPK.
4. Client deletes old SPK from device.

## Consequences

**Positive:**
- Server never holds private keys; cannot decrypt messages.
- OPK consumption ensures forward secrecy from session establishment.
- Multi-device support is first-class.

**Negative:**
- Multi-device increases message storage by O(devices).
- Key distribution API is a critical path; its availability determines whether new sessions can be established.
- OPK exhaustion (server ran out of OPKs for a user) falls back to no-OPK X3DH, which weakens forward secrecy.

## Interview Talking Points

- **Why OTK exhaustion is acceptable**: "Without a one-time pre-key, X3DH still works and provides forward secrecy at the session level via the Double Ratchet. It only loses the additional forward secrecy that OTKs add at session establishment. Signal documents this fallback explicitly."
- **Multi-device complexity**: "WhatsApp and Signal both handle multi-device by sending one ciphertext per device. This is why a WhatsApp group message to 256 members with 2 devices each results in 512 ciphertext copies in storage."
- **Why not a KMS (like AWS KMS)?** "KMS would mean the server derives or escrows keys. That defeats E2EE. The whole point is that the server never has key material."
